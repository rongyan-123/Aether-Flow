# 一、企业级规范开发步骤（纯逻辑无代码，全程按标准流程走，不会出现代码堆在一起的问题）
## 阶段1：项目初始化与架构规范搭建
**核心目标**：搭好标准化目录骨架，从根源避免后续迭代代码混乱
### 执行步骤
1. 用Vite官方脚手架初始化Vue3项目，不要手动搭建项目框架
2. 按前端行业标准创建目录结构，所有代码严格按类型放入对应文件夹，禁止把业务逻辑全部堆在App.vue单文件里
3. 配置Git版本管理，每个功能开发完成提交一次，提交信息统一按「feat: 完成XX功能」「fix: 修复XXbug」的规范写
### 需要掌握的最小知识点
- Vite初始化Vue3项目的基础命令
- Pinia的安装与基础配置方法
- Git基础的提交、推送操作
### 强制规范要求（第一版只保留以下必要目录，多余目录暂时不要加）
```
src/
├── apis/       # 所有接口请求逻辑统一放这里（比如豆包API的调用封装）
├── store/      # Pinia全局状态统一放这里（属性/任务/背包三个模块分开写）
├── components/ # 公共组件统一放这里（属性面板/背包面板/对话气泡等）
├── pages/      # 页面文件统一放这里（第一版只有1个游戏主页面）
├── utils/      # 通用工具函数统一放这里（本地存储封装/AI返回内容解析等）
├── App.vue     # 仅做页面路由/组件引入，不写具体业务逻辑
└── main.js     # 项目入口文件，仅做全局插件注册
```
> 第一版不要加TS、ESLint、单元测试等不熟悉的配置，避免过度设计卡壳，只要目录分层清晰就符合企业基础规范。

---
## 阶段2：公共底层能力封装
**核心目标**：把通用逻辑抽离，后续开发不用重复写代码，修改时只需要改一处
### 执行步骤
1. 封装本地存储工具函数：把读写属性/背包/任务数据的逻辑统一封装在utils文件夹里，所有组件要修改存储都调用这个封装好的函数，禁止直接写localStorage原生代码
2. 封装请求工具函数：把豆包API的通用请求头、密钥配置、错误处理逻辑统一封装在apis文件夹里，后续调用AI接口直接用封装好的方法，不要每次都写重复的请求逻辑
3. 配置环境变量：把豆包API密钥等敏感信息放在项目根目录的.env文件里，禁止写死在业务代码中
### 需要掌握的最小知识点
- localStorage的基础读写逻辑
- fetch/axios的基础请求逻辑
- Vite环境变量的基础配置规则
### 强制规范要求
.env敏感信息文件必须加到.gitignore里，绝对不能提交到Git公开仓库，避免密钥泄露被盗用，这个是企业开发的基础安全红线。

---
## 阶段3：三大核心功能模块开发（严格遵循单向数据流规范，后续AI读写数据不会混乱）
### 模块1：修仙面板模块
**核心解决问题**：保证数据修改可追溯，AI不会乱改面板数据
#### 执行步骤
1. 在store文件夹里分别写属性/任务/背包三个独立的状态模块，所有数据修改只能通过store的actions方法，禁止直接修改state
2. 分别开发三个面板组件，数据绑定对应store的state，操作按钮调用对应store的actions方法
3. 联调本地存储：页面加载时从localStorage读取数据初始化store，数据变化时自动同步到localStorage
#### 规范要求
严格遵循Vue单向数据流规范，所有数据修改入口唯一，后续AI要改数据也直接调用store的actions，不会出现数据混乱的问题。

---
### 模块2：AI对话交互模块
**核心解决问题**：实现上下文连贯、流式输出的对话体验
#### 执行步骤
1. 开发对话界面组件：区分用户消息/AI消息样式，实现新消息发送后自动滚动到底部的逻辑
2. 调用封装好的豆包API，实现流式输出渲染（不用等AI全部返回再显示，逐字输出体验更好）
3. 把对话历史存在单独的store模块里，每次请求AI时带上最近10轮的历史消息，保证剧情上下文连贯
#### 规范要求
对话历史和面板数据分开存储，不要混在同一个store模块里，后续扩展多轮剧情不会冲突。

---
### 模块3：双向数据同步模块
**核心解决问题**：让AI准确识别数据修改指令，不会乱改面板数据
#### 执行步骤
1. 写固定的prompt规则，明确告诉AI必须返回约定的结构化格式：比如剧情内容放在content字段，要修改的数据放在modify字段里，禁止返回纯文本
2. 开发AI返回内容解析工具函数，先校验返回格式是否符合要求，不符合就重新请求AI
3. 加数据修改校验：如果解析到modify字段，先判断修改的字段是否存在、数值是否合法（比如修为不能为负、背包不能超上限），校验通过再调用store的actions修改数据
#### 规范要求
所有数据修改必须经过校验层，避免AI返回错误格式导致页面崩溃，这个是企业级数据交互的标准容错逻辑。

---
## 阶段4：联调测试与上线前校验
**核心解决问题**：保证上线后没有致命bug
### 执行步骤
1. 跑通全流程测试用例：打开页面初始数据正常→发消息AI返回对应剧情→选选项后AI正确读取当前属性生成匹配内容→AI修改数据后面板实时更新→刷新页面数据不丢失
2. 打包测试：执行build命令打包，本地预览dist文件夹有没有报错，避免打包后白屏
### 规范要求
第一版只测核心流程，不用测边缘场景，符合最小闭环要求。

# 二、上线发布流程科普
## 方案1：第一版优先选免费静态托管（0成本、不用买服务器、符合企业CI/CD规范）
适合当前纯前端、数据存在用户本地的最小版本，流程如下：
1. 代码上传：把项目代码推送到Github/Gitee的公开仓库（确保.env文件已经加到.gitignore里，没有提交敏感信息）
2. 托管平台注册：用Github账号直接登录Vercel/Netlify（国际版不用备案，国内可选Vercel国内镜像或者阿里云OSS静态托管）
3. 自动部署：在托管平台后台导入你的Github仓库，点击部署，平台会自动执行打包、部署流程，2-5分钟完成
4. 环境变量配置：在托管平台后台的环境变量配置页，把本地.env里的豆包API密钥填进去，避免密钥泄露
5. 上线访问：部署完成后平台会给你一个免费的二级域名，直接就能访问；后续你往Github推新代码，平台会自动更新线上版本，完全符合企业级自动部署规范。

---
## 方案2：后续迭代可选云服务器部署（适合加了后端、多用户系统之后用）
如果你后续要做多用户登录、云存储数据等功能，才需要买服务器，流程如下：
1. 资源采购：买阿里云/腾讯云的轻量应用服务器（新人一年几十块），如果要自定义域名可以买个域名（几块钱一年，国内需要备案）
2. 服务器配置：服务器上安装Nginx，配置静态资源路径，把打包好的dist文件夹传到服务器对应的目录
3. 域名解析：把你的域名解析到服务器的公网IP，配置SSL证书开启HTTPS
4. 后端部署：如果有后端接口，把后端服务也部署到服务器，用Nginx做反向代理，把接口请求转发到后端服务
5. 上线校验：访问域名测试所有功能正常，配置日志监控方便后续排查问题。

# 三、关键执行注意事项
1. 不要过度设计：第一版不要加任何非必要的配置/功能，只要目录分层清晰、核心流程跑通就够，避免复杂度太高半途而废放大凶性
2. 严格遵守安全规范：敏感信息绝对不能提交到公开仓库，这个是企业开发的红线，避免造成损失
3. 版本管理一定要做：每个功能开发完提交一次Git，出问题可以随时回滚，这个是放到简历里也能加分的好习惯
4. 上线后只要有10个以上真实用户试用反馈，就算完成木生火的闭环，不用追求完美，先上线再迭代。
